## Prerequisites

- A running Kubernetes cluster.
- Helm 3 or above. See Installing [Helm](https://helm.sh/docs/intro/install).
- Access to a Postgres instance (or use the default Postgres chart installed by Convoy's Helm chart)
- Access to a Redis instance (or use the default Redis chart installed by Convoy's Helm chart)

<Note>
	The Convoy Helm chart depends on Bitnami Postgres and Redis charts and will install them by default for testing and evaluation. For production, you should use managed or
	self-hosted Postgres and Redis instead of these defaults and configure the chart to point at your external databases.
</Note>

## Steps

Add Convoy’s chart repository to Helm:

```shell
helm repo add convoy https://frain-dev.github.io/helm-charts
```

Update the chart repository:

```shell
helm repo update
```

Install the chart with default values:

```shell
helm install convoy convoy/convoy --namespace convoy --create-namespace
```

Install the chart with a custom `values.yaml`:

```shell
helm install convoy convoy/convoy --namespace convoy --create-namespace --values values.yaml
```

Upgrade the chart:

```shell
helm upgrade convoy convoy/convoy --namespace convoy --values values.yaml
```

## Upgrading

### Upgrading to Convoy v24.8.x (Convoy Helm charts v3.1.0)

To upgrade to Convoy [v24.8.x](../../changelog/releases#v24-8-x),
you need to update the chart to the `3.x.x` series.
See the [Convoy Helm chart on ArtifactHub](https://artifacthub.io/packages/helm/convoy/convoy) for the latest `3.x.x` version.
In this release, we have gated a number of features behind a license.
If you use them, these features will cease to work until you provide
a license key.
Read [here](/business-and-enterprise/licensing) to learn about all the paid features,
and [here](/business-and-enterprise/paid-features) to learn how to get a license key.

### Upgrading the Convoy version without upgrading the chart

Change the image tag values to the Convoy version you want. The chart exposes a **global tag** that fans out to all Convoy components (server and agent), and **per-component tags** if you need finer control:

```yaml values.yml
# Recommended: update all Convoy components at once
global:
    convoy:
        tag: vX.Y.Z # replace with the Convoy version you want to run (for example v26.1.4; see Convoy Releases for the latest version)
```

## Production deployment (managed Postgres and Redis)

For production, we strongly recommend using managed Postgres and Redis instead of the bundled Bitnami charts.

Disable the bundled Postgres and Redis charts and switch to external services:

```yaml values.yml
postgresql:
    enabled: false

redis:
    enabled: false

global:
    externalDatabase:
        enabled: true # use an external / managed Postgres
    nativeRedis:
        enabled: false # do not deploy the in-cluster Redis
    externalRedis:
        enabled: true # use an external / managed Redis
```

### Example: managed Postgres (RDS, CloudSQL, etc.)

Point Convoy at a managed Postgres instance by configuring `global.externalDatabase` and disabling the in-cluster Postgres chart:

```yaml values.yml
global:
    externalDatabase:
        enabled: true
        host: my-postgres-host.rds.amazonaws.com
        port: 5432
        database: convoy
        username: convoy
        # When 'secret' is set, the inline password values are ignored.
        # Create a Kubernetes Secret with key 'password' containing your DB password.
        secret: convoy-postgres
        # options: "sslmode=require&connect_timeout=30" # example for managed Postgres with TLS

postgresql:
    enabled: false
```

### Example: managed Redis

Use a managed Redis (e.g. AWS ElastiCache, GCP Memorystore, Azure Managed Redis, etc.) by configuring `global.externalRedis`, disabling `global.nativeRedis`, and disabling the in-cluster Redis chart:

```yaml values.yml
global:
    nativeRedis:
        enabled: false
    externalRedis:
        enabled: true
        host: my-redis-host.cache.amazonaws.com
        port: '6379'
        scheme: 'rediss' # use 'rediss' for TLS-enabled Redis endpoints
        database: '0'
        # When 'secret' is set, the inline password values are ignored.
        # Create a Kubernetes Secret with key 'password' containing your Redis password.
        secret: convoy-redis

redis:
    enabled: false
```

### Connectivity and startup behaviour

- If Postgres or Redis are unreachable (DNS, firewall, wrong credentials, TLS issues), Convoy pods will fail to start and may go into `CrashLoopBackOff`. Check the pod logs for connection errors.
- Helm itself will succeed in creating resources, but the deployment will not become healthy until Convoy can connect to the database and Redis.
- For managed services with TLS, ensure you set appropriate connection options (for Postgres) or `scheme: rediss` (for Redis), and that your network policies / security groups allow traffic from the cluster to the managed service.
- Both the server and agent deployments include a `wait-for-migrate` init container that runs `migrate up` before the application starts. If migrations cannot run (for example, the database is unreachable), these init containers will fail and the pods will not become Ready.

## Scaling and resources

Use `server.app.resources` and `agent.app.resources` to set CPU/memory requests and limits, and `server.autoscaling` / `agent.autoscaling` to enable horizontal autoscaling.

Start with conservative values and tune based on your event volume and observed CPU/memory usage:

```yaml values.yml
server:
    app:
        resources:
            requests:
                cpu: 500m
                memory: 1Gi
            limits:
                cpu: 1
                memory: 2Gi
    autoscaling:
        enabled: true
        minReplicas: 2
        maxReplicas: 10
        targetCPUUtilizationPercentage: 70
        targetMemoryUtilizationPercentage: 80

agent:
    app:
        resources:
            requests:
                cpu: 500m
                memory: 1Gi
            limits:
                cpu: 1
                memory: 2Gi
    autoscaling:
        enabled: true
        minReplicas: 2
        maxReplicas: 10
        targetCPUUtilizationPercentage: 70
        targetMemoryUtilizationPercentage: 80
```

For higher traffic workloads, increase the requests/limits and allow higher `maxReplicas`, and use metrics from your cluster (Prometheus, cloud monitoring) to fine-tune thresholds.

## Path prefix and Ingress {#path-prefix-and-ingress}

You can expose the **server** (control plane) and **agent** (data plane) on **one hostname** with different path prefixes—for example `https://webhooks.example.com/server` and `https://webhooks.example.com/agent`.

Convoy uses the same setting for both processes: **`CONVOY_ROOT_PATH`** (or `root_path` in `convoy.json`). Each deployment must use the prefix that matches what your Ingress sends to that Service (no trailing slash; must start with `/`). See [Root path](/deployment/configuration#root-path) for format rules and behaviour.

### Helm values

Set the root path **per component** and avoid duplicate Ingress rules for the same host (disable the subchart Ingress objects if you manage routing yourself):

```yaml values.yml
server:
    ingress:
        enabled: false
    env:
        root_path: '/server'

agent:
    ingress:
        enabled: false
    env:
        root_path: '/agent'
```

For **separate hostnames** (for example `server.example.com` and `agent.example.com`), leave `server.env.root_path` and `agent.env.root_path` empty and use `server.ingress` / `agent.ingress`, or your own Ingress with path `/` per host.

### Example Ingress

Point each prefix at the Kubernetes Services created by the chart. Default Service names are **`convoy-server`** and **`convoy-agent`** (from `server.fullNameOverride` / `agent.fullNameOverride`); Service ports are usually **`server.service.port`** and **`agent.service.port`** (often `80`).

```yaml Ingress
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
    name: convoy-paths
    namespace: convoy
spec:
    ingressClassName: nginx # replace with your ingress class
    rules:
        - host: webhooks.example.com
          http:
              paths:
                  - path: /server
                    pathType: Prefix
                    backend:
                        service:
                            name: convoy-server
                            port:
                                number: 80
                  - path: /agent
                    pathType: Prefix
                    backend:
                        service:
                            name: convoy-agent
                            port:
                                number: 80
```

Adjust `namespace`, `ingressClassName`, host, paths, and ports to match your release. Add TLS and controller-specific annotations as needed. The proxy should forward the **full path** to the pod (no rewrite that strips `/server` or `/agent` unless you intentionally configure the app for that).

<Note>
	When the control plane has a non-empty root path, Convoy also registers API routes at `/` to support the dashboard; restrict what is publicly reachable using your Ingress or
	network policies if you rely on path isolation.
</Note>

## ArgoCD Configuration

Convoy uses helm hooks to trigger migrations every time the charts are installed or upgraded. These hooks alone won't work for an ArgoCD installation since they are not compatible.

```yaml Helm hooks
'helm.sh/hook': post-install,post-upgrade
'helm.sh/hook-delete-policy': before-hook-creation
```

You can apply this ArgoCD definition to run migrations on your cluster when the chart is installed and upgraded

```yaml ArgoCD Definition
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
    name: convoy
    namespace: argocd
    finalizers:
        - resources-finalizer.argocd.argoproj.io
spec:
    project: default
    source:
        repoURL: 'https://frain-dev.github.io/helm-charts'
        targetRevision: 'X.Y.Z' # use the chart version you want to deploy (for example 3.7.5; see ArtifactHub for the latest version)
        chart: convoy
        helm:
            releaseName: convoy
            valuesObject:
                migrate:
                    jobAnnotations:
                        argocd.argoproj.io/hook: Sync
    destination:
        server: 'https://kubernetes.default.svc'
        namespace: convoy
    syncPolicy:
        syncOptions:
            - ServerSideApply=true
            - CreateNamespace=true
        automated:
            selfHeal: true
            prune: true
```
