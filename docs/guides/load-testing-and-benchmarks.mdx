This page supports quality and capacity conversations (for example enterprise onboarding): what to measure when stress-testing Convoy, and how we publish benchmark results.

## What to measure

Focus on the paths that match your deployment:

1. **Ingestion** — HTTP ingest (or broker ingest) sustained rate, error rate, and latency to accept an event.
2. **Delivery** — Successful delivery rate to downstream endpoints, retry behaviour, and [end-to-end latency](/product-manual/metrics) (`convoy_end_to_end_latency` when Prometheus is enabled).
3. **Datastore and Redis** — Postgres CPU, connections, and slow queries; Redis memory and command latency under queue load.
4. **Agent capacity** — Horizontal scale of `convoy agent` replicas vs. backlog growth.

Record Convoy version, hardware or k8s limits, and whether features such as [read replicas](/product-manual/read-replicas), [circuit breaker](/product-manual/circuit-breaker), or [retention](/product-manual/retention-policy) are enabled.

## How to run load tests

- Drive **API traffic** with your preferred tool (k6, Locust, Gatling, `vegeta`, etc.) against the ingest URL and, if needed, control-plane APIs.
- For **webhook consumers**, use a pool of HTTP servers (or a mock service) that returns configurable status codes and latency so you can test retries and failures.
- Enable **[Prometheus metrics](/product-manual/metrics)** on a staging cluster and scrape `/metrics` during the test window.
- Prefer **gradual ramp** (stepped RPS) until error budgets or latency SLOs break; capture the last stable point as a rough ceiling for that topology.

## Sizing and capacity (guidance)

Convoy does **not** publish a single “events per second” or “events per month” figure that fits every tenant: throughput depends on **agent replica count**, **Postgres and Redis** headroom, **network RTT** to subscribers, **payload size**, **retry volume**, and features such as [read replicas](/product-manual/read-replicas) or [circuit breaker](/product-manual/circuit-breaker). Use this page’s methodology plus [Metrics](/product-manual/metrics) on a staging cluster to find the ceiling for *your* topology. Official benchmark summaries for specific scenarios are shared under NDA or by sales engineering when available—see below.

**Multi-project / multi-tenant:** Organisations and projects are the primary isolation boundaries in the product model; see [Organisations and projects](/product-manual/organizations-and-projects). Operational isolation (compute, data residency, noisy-neighbour behaviour) still depends on how you deploy and shard instances.

## Payload and request size limits

- **Ingest body limit:** Convoy enforces a configurable maximum request body size for ingestion: JSON key **`max_response_size`**, env **`CONVOY_MAX_RESPONSE_SIZE`**. The stored value is interpreted as **kilobytes** and converted to bytes at startup (default **50** KiB → **51200** bytes unless overridden). Confirm behaviour on your exact release in the Convoy `Configuration` struct in source.
- **Reverse proxies and ingress:** Load balancers, API gateways, and Kubernetes ingress controllers often impose their own body limits—set them **≥** Convoy’s limit to avoid opaque 413s at the edge.
- **Downstream delivery:** Subscriber response bodies are read with an internal limit during dispatch (bounded read); enormous payloads still stress DB and network—measure end-to-end.

## Load test report template

When sharing results internally or with Convoy, include:

- Convoy **version**, **license** tier (if relevant), and feature flags enabled
- Topology: **server** and **agent** replicas, **Postgres** and **Redis** specs (or cloud instance types)
- **Ingest** path: HTTP vs broker; sustained **RPS**, **error rate**, p50/p95 **accept latency**
- **Delivery**: success rate to test endpoints, retry behaviour, **p95** from `convoy_end_to_end_latency` if Prometheus is enabled
- **Datastore**: Postgres CPU, connections, slow queries; Redis memory and latency
- **Queue metrics:** Redis and Postgres gauges from [Metrics](/product-manual/metrics) at peak load
- Payload **size distribution** (typical and worst case)

## Published benchmark results

Internal throughput and latency numbers are shared with customers under NDA or in sales engineering decks as they are refreshed. If you need a specific scenario (region, HA, broker-backed ingest), ask your Convoy contact and we will align with engineering on a reproducible profile.

When we publish summary numbers publicly, they will be linked from this page and from the [changelog](/changelog/core-gateway).
